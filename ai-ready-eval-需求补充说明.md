# AI-Ready Eval — 需求补充说明

> 针对 Spec v2 的核心功能澄清
> 
> 重点：Skill 评估和 API AI-Ready 测试的输入形态与检测方法

---

## 当前问题一句话总结

/run-eval 页面只有一个"上传 OpenAPI 规范"的入口，暴露了两个根本性的理解偏差：

1. **Skill 评估不是"上传一个 markdown 文件做静态文本分析"**——Skill 是一个包含多个文件的工程包，评估它需要让真实 agent 加载并执行它
2. **API AI-Ready 测试不是"解析一个 OpenAPI spec 数数有没有 description"**——真正的 AI-Ready 需要让真实 agent 去调用接口，看它能不能完成任务

两个功能的共同点是：**评估的裁判必须是真实的 AI agent，不是静态文本分析器。** 静态分析只是预检，不是终裁。

## 功能一：Skill 评估与调优

### 什么是 Skill

Skill 不是一个单独的文件。它是一个文件包，典型结构如下：

```
my-skill/
├── SKILL.md                    # 主文件：指令、触发条件、核心规则
├── references/                 # 参考文档：详细规范、术语表、模板
│   ├── report-structure.md
│   ├── data-governance.md
│   └── api.md
├── scripts/                    # 脚本：确定性任务的自动化代码
│   └── validate.py
├── assets/                     # 资源：模板、图标、字体等
│   └── template.docx
└── evals/                      # 评估用例（如果已有）
    ├── evals.json
    └── trigger_evals.json
```

用户的 skill 可能来自不同渠道，格式不统一。我们需要兼容多种载入方式。

### Skill 载入方式（按优先级）

**方式一：上传文件夹**
- 用户选择本地文件夹（利用浏览器的 `webkitdirectory` 属性），系统读取整个目录结构
- 这是最完整的方式，保留了所有文件层次关系

**方式二：上传压缩包**
- 支持 .zip / .tar.gz
- 前端解压后还原目录结构
- 适合从 GitHub 下载的 skill 包

**方式三：上传单个 SKILL.md**
- 最简模式——用户只有一个 markdown 文件
- 系统自动识别为"无 references、无 scripts"的简单 skill
- 评估时标注"references 缺失，部分维度无法评估"

**方式四：粘贴 SKILL.md 内容**
- 最低门槛——复制粘贴到文本框
- 同方式三，按单文件模式处理

**方式五：Git 仓库地址**（进阶功能，可后续实现）
- 输入 GitHub/GitLab 仓库 URL
- 后端 clone 并解析

**方式六：npm/pip 包名**（进阶功能，可后续实现）
- 输入包名，后端 install 并定位 SKILL.md
- 支持 Anthropic 的 .skill 打包格式

### 载入后的统一数据结构

无论哪种载入方式，系统内部统一转换为：

```json
{
  "skill_name": "investment-research",
  "files": {
    "SKILL.md": "---\nname: investment-research\n...",
    "references/report-structure.md": "# 报告结构详细指南\n...",
    "references/api.md": "# HTTP API 参考文档\n...",
    "scripts/validate.py": "import json\n...",
    "evals/evals.json": "{\"skill_name\": ...}"
  },
  "metadata": {
    "载入方式": "文件夹上传",
    "文件总数": 6,
    "SKILL.md行数": 347,
    "有references目录": true,
    "有evals目录": true,
    "有scripts目录": true
  }
}
```

### Skill 评估的三个层次

#### 层次一：结构与内容分析（纯前端，秒级）

这部分**可以**做静态分析，对应 spec 中的"结构合规"和"指令质量"两个维度：

| 检查内容 | 方法 | 不需要 AI |
|---------|------|----------|
| SKILL.md 行数是否合理 | 数行数 | ✓ |
| frontmatter 是否完整 | 解析 YAML | ✓ |
| 是否有 references 分层 | 检查目录结构 | ✓ |
| 命令式规则密度 | 正则匹配"禁止/必须/MUST" | ✓ |
| 是否有示例 | 搜索"示例/Example/如" | ✓ |
| 是否有 evals 文件 | 检查文件存在性 | ✓ |

**这一层的价值**：快速给用户一个初始诊断。但必须明确标注"这是结构检查，不是功能验证"。

#### 层次二：触发准确率测试（需要 LLM，分钟级）

测试 skill 的 description 字段是否能让 agent 正确决定"要不要用这个 skill"。

**输入**：
- skill 的 name + description（从 frontmatter 提取）
- 一组 trigger eval 用例（如果用户 skill 包里有 trigger_evals.json 就用，没有则系统自动生成）

**自动生成 trigger eval 的方法**：
1. 用 LLM 阅读 SKILL.md 的完整内容
2. 让 LLM 生成 10 个"应该触发"的用户 prompt（覆盖不同措辞和场景）
3. 让 LLM 生成 10 个"不应该触发"的 near-miss prompt（和 skill 功能相近但实际不该触发）
4. 对每个 prompt，让另一个 LLM 实例仅基于 description 判断是否应该触发
5. 计算准确率

**这一层的价值**：验证 skill 的"入口"是否准确。触发不准的 skill，后面的功能再好也白搭。

#### 层次三：功能可靠性测试（需要 LLM + 可能需要工具，分钟级到小时级）

这是最核心、最有价值的测试——**让真实 agent 加载这个 skill 并执行任务**。

**执行方式**：
1. 把 SKILL.md 内容 + references 内容组装成 system prompt
2. 用 eval 用例中的 prompt 作为 user message
3. 让 LLM（如 Claude）按照 skill 指令执行任务
4. 检查输出是否满足 evals.json 中的 assertions
5. 如果 skill 依赖外部工具（API 调用），有两种模式：
   - **模拟模式**：用 mock 响应替代真实 API 调用
   - **实战模式**：用户提供真实 API 凭证，agent 实际调用

**如果用户没有 evals.json**：
1. 让 LLM 阅读 SKILL.md，理解 skill 的目标
2. 自动生成 3-5 个测试用例和对应的验收标准
3. 提交给用户确认后执行

**这一层的价值**：这是唯一能回答"这个 skill 实际上好不好用"的方法。

### Skill 调优建议

评估完成后，系统不只是给分数，还要给**具体的、可执行的改进建议**：

| 问题类型 | 建议格式 |
|---------|---------|
| SKILL.md 过长 | "建议将第 120-180 行的数据治理规则移到 references/data-governance.md，在主文件中添加一行引导：'生成报告前读取 references/data-governance.md'" |
| 触发描述太窄 | "当前 description 未覆盖以下场景：[具体 prompt]。建议在 description 中添加：'即使用户没有明确说 XX，只要问题涉及 YY，都应触发本 skill'" |
| 命令式规则过多 | "第 45 行'禁止出现无来源评分'建议改为解释式：'投资经理需要基于原始事实判断，AI 合成评分无法追溯依据，因此报告中的量化应引用可追溯的一手数据'" |
| 无 baseline 对比 | "建议创建 evals/ 目录，添加 3 个核心场景的测试用例，运行一次带 skill 和不带 skill 的对比" |

## 功能二：API AI-Ready 测试

### 根本思路转变

上一版把"AI-Ready API 测试"做成了"解析 OpenAPI spec 文件的静态检查器"。这只是预检环节。

**真正的 AI-Ready 测试是这样的**：给一个 AI agent 一个任务和一个 API 的说明，看它能不能完成任务。能完成 = API 对 agent 友好；完不成 = API 需要改造。

这意味着测试的输入不只是一个 spec 文件，而是：

### API 测试的输入形态

用户需要提供两样东西：

**第一：API 的描述信息**（让 agent 知道有什么接口可用）

| 载入方式 | 说明 | 适用场景 |
|---------|------|---------|
| 上传 OpenAPI spec（YAML/JSON） | 最标准的方式 | 已有规范文档的 API |
| 粘贴接口说明文本 | 自然语言描述接口——端点、参数、返回值 | 没有正式 spec 的内部 API |
| 输入 MCP Server 地址 | 系统自动获取 tool 列表和描述 | 已封装为 MCP 的 API |
| 输入 API 文档 URL | 爬取文档页面提取接口信息 | 有在线文档但没有 spec 文件 |

**第二：可调用的 API 地址 + 认证**（让 agent 能实际调用）

```
基础地址: https://api.example.com
认证方式: API Key
认证值: sk-xxxx（或 Bearer Token / OAuth2）
```

没有这个，就只能做静态分析（"你的 spec 写得怎么样"），不能做真正的 AI-Ready 测试（"agent 能不能用你的 API"）。

### API 测试的三个递进层次

#### 层次一：Spec 静态分析（无需调用 API，无需 LLM）

这是现在已实现的部分，但需要**明确标注这只是预检**。

检查的是"你的接口描述文档写得对 agent 友好吗"，不是"你的接口对 agent 友好吗"。

保留 8 维度静态检查，但在结果页面顶部显示：

> ⚠️ 当前为静态分析模式——基于接口描述文档的文本分析。
> 如需验证 API 实际行为是否对 AI 友好，请提供 API 地址和认证信息，运行动态测试。

#### 层次二：动态行为探测（需要调用 API，不需要 LLM）

用程序化的方式调用 API，验证其行为特征是否对 agent 友好。

**这一层回答的问题**：API 的实际行为和 spec 描述是否一致？实际响应大小是多少？错误信息是否可解读？限流行为是否合理？

| 测试类别 | 具体操作 | 检查什么 |
|---------|---------|---------|
| **Spec-实现一致性** | 对 spec 中每个端点发真实请求 | 端点是否存在、返回结构是否匹配 schema、必填参数缺失时是否报错 |
| **响应体量实测** | 对列表端点发无参数请求 | 默认返回多少条、多少字节、估算多少 token |
| **分页验证** | 分别传 page=1&limit=5 和 page=2&limit=5 | 分页是否工作、结果是否不重叠 |
| **错误语义实测** | 故意发错误请求（缺参数、类型错、无权限） | 错误响应是否结构化、是否指出具体问题、是否有修复建议 |
| **限流行为** | 短时间发 10-20 个请求 | 是否返回 429、是否有 Retry-After、恢复后是否正常 |
| **幂等性验证** | 对同一 POST 端点发两次相同请求 | 是否创建了两份资源（无幂等性）还是只创建一份 |

#### 层次三：Agent 实战测试（需要调用 API + 需要 LLM）🔑

**这是整个产品最核心的价值点**——别的工具做不到的地方。

**原理**：
1. 把 API 的描述信息（spec / 文档 / MCP tool list）提供给 LLM 作为工具定义
2. 给 LLM 一个业务任务（用户指定或系统自动生成）
3. 观察 LLM 是否能正确调用 API 完成任务
4. 记录调用链中每一步的成功/失败/错误，分析 root cause

**执行流程**：

```
输入：API spec + API 地址 + 认证 + 任务列表
                │
                ▼
        为 LLM 构造工具定义
        （从 spec 提取每个端点 → function calling 格式）
                │
                ▼
        对每个测试任务：
                │
                ├→ 让 LLM 规划调用步骤
                ├→ LLM 发出工具调用请求
                ├→ 系统代理执行 HTTP 调用
                ├→ 将响应返回给 LLM
                ├→ LLM 决定下一步或完成
                │
                ▼
        记录完整调用链：
        {
          "任务": "查询所有进行中的任务并按优先级排序",
          "步骤": [
            {
              "步骤号": 1,
              "agent动作": "调用 GET /tasks?status=active",
              "是否正确": true,
              "实际响应": "200, 返回 15 条记录, 4.2KB",
              "agent解读": "正确解析了返回的任务列表"
            },
            {
              "步骤号": 2,
              "agent动作": "尝试用 sort_by=priority 参数重新查询",
              "是否正确": false,
              "失败原因": "API 不支持 sort_by 参数，但 spec 中未说明排序方式",
              "根因分类": "语义描述不足——spec 未描述如何实现排序"
            }
          ],
          "任务完成": false,
          "失败根因": "语义描述不足",
          "改进建议": "在 GET /tasks 端点的 description 中说明支持的排序参数和默认排序规则"
        }
```

**每个失败步骤的根因分类**（直接映射到 8 个评估维度）：

| 根因 | 对应维度 | 含义 |
|------|---------|------|
| 语义描述不足 | 维度 1 | Agent 不知道端点/参数的用途，调错了 |
| 响应体量过大 | 维度 2 | 返回数据太多，agent 后续推理出错 |
| 错误不可解读 | 维度 3 | API 报错了但 agent 不知道怎么修 |
| 端点找不到 | 维度 4 | Spec 里有但实际 404，或 spec 里没有 |
| 不知道下一步 | 维度 5 | 多步任务中 agent 不知道该调哪个端点 |
| 参数格式猜错 | 维度 6 | 其他端点用 snake_case 这个端点用 camelCase |
| 被限流中断 | 维度 7 | 任务执行到一半被 429 打断 |
| 权限不足 | 维度 8 | 需要的操作没有对应权限 |

**测试任务的来源**：

1. **用户指定**：用户在界面输入"我希望 agent 能用这个 API 完成以下任务：..."
2. **系统自动生成**：LLM 阅读 spec，推断 3-5 个典型业务场景：
   - 一个简单查询任务（单端点）
   - 一个带筛选/排序的查询任务（参数使用）
   - 一个创建+查询的多步任务（工作流）
   - 一个错误恢复任务（故意触发错误看 agent 能否自愈）
3. **用户确认**：自动生成的任务提交给用户确认/修改后再执行

### API 测试结果的展示

不只是给一个 A/B/C/D 的分数，而是展示：

**总览**：综合评分 + 任务完成率（3/5 个任务完成）+ 最常见的失败根因

**任务回放**：对每个测试任务，展示 agent 的完整"思考和行动"过程——它尝试了什么、成功了什么、失败了什么、失败原因是什么。用户可以像看"录像回放"一样看 agent 和 API 交互的全过程。

**维度诊断**：8 个维度的分数，但分数来源不再是"数了几个 description"，而是"在实际任务中这个维度导致了几次失败"。比如"语义描述完整性: 60%"不是因为"40% 参数没有 description"，而是因为"5 个测试任务中，有 2 个因为描述不足导致 agent 调错了端点"。

**改进路径**：按"修复后对任务完成率的提升幅度"排序的建议列表。

## 两个功能的交汇点：Skill + API 联合测试

最有价值的场景是同时测试：一个 Skill 依赖了哪些 API，这些 API 对 agent 来说够不够用。

**场景**：用户上传了一个投研分析 Skill，这个 Skill 里引用了论文数据库 API、新闻数据库 API、人才数据库 API。联合测试可以回答：

- 这个 Skill 的指令写得够不够清晰？（Skill 评估）
- 它依赖的 API 对 agent 够不够友好？（API 评估）
- 端到端——agent 加载这个 Skill 并调用这些 API，能不能完成一个完整的投研任务？（联合测试）

**这才是系统的终极形态——不是分别给 Skill 和 API 打分，而是验证"AI 员工能不能上岗"。**

## 界面重新设计

### 页面结构

整个应用只需要 4 个主页面：

| 页面 | 路径 | 功能 |
|------|------|------|
| 首页 | / | 产品介绍 + 两个入口（Skill 评估 / API 测试） |
| Skill 评估 | /skill-eval | 上传 Skill → 三层评估 → 结果展示 → 调优建议 |
| API 测试 | /api-eval | 载入 API → 三层测试 → 结果展示 → 改进路径 |
| 联合测试 | /integration | 同时载入 Skill + API → 端到端测试 |

不需要单独的"趋势页"（评估历史放在各自页面的底部）。
不需要单独的"概览仪表盘"（首页就是入口，不堆 mock 数据）。

## 实现优先级

考虑到开发资源有限，建议按以下顺序交付：

### P0：最小可用版本（让产品能用）

1. **Skill 评估 — 层次一（结构分析）**：上传 SKILL.md → 结构检查 + 指令质量分析 → 显示结果
2. **API 测试 — 层次一（Spec 静态分析）**：上传 OpenAPI spec → 8 维度静态检查 → 显示结果
3. **基本 UI**：深色主题、全中文、维度卡片展开/收起

> P0 不需要 LLM，不需要真实 API 调用。纯前端就能跑。

### P1：核心差异化（让产品有价值）

4. **API 测试 — 层次三（Agent 实战）**：输入 API 地址 + LLM Key → agent 执行任务 → 调用链回放 → 根因分析
5. **Skill 评估 — 层次二+三（触发 + 功能测试）**：输入 LLM Key → 触发准确率测试 + 功能测试

> P1 是产品的核心卖点——"让真实 AI 去用你的 API / Skill，告诉你哪里有问题"。别的工具做不到这一点。

### P2：完整体验

6. **Skill 多文件载入**：文件夹上传、压缩包上传
7. **API 测试 — 层次二（动态探测）**：自动化行为测试
8. **联合测试**：Skill + API 端到端
9. **报告导出**：JSON + 独立 HTML
10. **评估历史 + 趋势**

### P3：进阶功能

11. MCP Server 地址载入
12. Git 仓库载入
13. npm/pip 包载入
14. CI/CD 集成
15. 调优建议自动应用（一键修复）
